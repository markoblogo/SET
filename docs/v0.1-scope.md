# SET v0.1 Scope

This document defines the first version of the orchestrator.

## Goal

Ship one stable GitHub Action entrypoint that orchestrates repo-facing tooling without reimplementing tool logic.

## v0.1 principle

Keep `SET` thin.

`SET` should install tools and run them in a predictable order.
It should not absorb business logic from `agentsgen`, `git-tweet`, or future tools.

## What v0.1 includes

### Supported tool family

Only `agentsgen` in v0.1.

### Supported operations

- install `agentsgen`
- run `agentsgen init`
- optionally run `agentsgen pack`
- optionally run `agentsgen pack --site <url>`
- optionally run `agentsgen check --all --ci`
- optionally run `agentsgen understand --compact-budget <tokens>` with optional `--focus` / `--changed` slice flags
- optionally run `agentsgen snippets`
- optionally run `agentsgen analyze <url>`
- optionally run `agentsgen meta <url>`

### Initial inputs

Suggested minimal input surface:

```yaml
with:
  agentsgen: ""
  init: ""
  pack: ""
  site_pack: ""
  site_url: ""
  check: ""
  repomap: ""
  repomap_compact_budget: "4000"
  repomap_focus: ""
  repomap_changed: ""
  snippets: ""
  analyze: ""
  analyze_url: ""
  meta: ""
  meta_url: ""
  autodetect: "true"
  workflow_preset: ""
  preset: ""
  path: "."
```

Notes:
- `agentsgen` is the feature family switch
- `workflow_preset` sets a baseline such as `minimal`, `repo-docs`, or `site-ai`
- `preset` is optional and only affects `agentsgen init --preset`
- `site_url` is only required when `site_pack` is enabled
- `repomap_compact_budget` defaults to `4000` when `repomap` is enabled; `repomap_focus` / `repomap_changed` are optional slice controls
- SET surfaces derived repomap policy modes in docs/planning/summary: `full` (Full Repo Slice), `focus` (Focused Code Slice), `changed` (Changed Files Slice), `focus+changed` (Hybrid Slice)
- `analyze_url` is only required when `analyze` is enabled
- `meta_url` is only required when `meta` is enabled
- `path` defaults to repo root

## What v0.1 explicitly does not include

- no `git-tweet` orchestration
- no standalone `LLMO` service/runtime inside `SET`
- no live dashboard backend
- no direct GitHub API repo mutation
- no automatic PR/apply flow
- no multi-tool dependency graph beyond a simple ordered flow

## Execution order

The default execution order should be:

1. install runtime
2. run `agentsgen init` if enabled
3. run `agentsgen pack` if enabled
   when `site_pack` is enabled, the same pack step runs with `--site <url>`
4. run `agentsgen check --all --ci` if enabled
5. run `agentsgen understand` if enabled
6. run `agentsgen snippets` if enabled
7. run `agentsgen analyze <url>` if enabled
8. run `agentsgen meta <url>` if enabled

## Why this scope

This gives one working entrypoint quickly while keeping risk low.

It also creates the stable place where future tools can be added one by one.

## v0.1 success criteria

- one reusable Action in `SET`
- works in a clean target repo with only inputs
- no duplicated `agentsgen` logic in `SET`
- output is readable in GitHub Actions logs
- writes a compact GitHub Actions summary for the resolved plan
- adding a second tool later does not require redesign

## Next after v0.1

### v0.2 candidates

- registry-driven tool definitions
- structured execution summaries
- support for registry-backed tool metadata

### later

- `git-tweet` integration
- automated PR/apply flow
- non-agentsgen tool integrations
- live dashboard backend

## Presets

`SET` v0.1 now supports small orchestration presets.

- `minimal` -> initialize repo docs only
- `repo-docs` -> initialize + pack + check
- `site-ai` -> repo-docs + site pack + analyze + meta

Rule: explicit action inputs override preset defaults.

- optional proof-loop hook (`agentsgen task init/evidence/verdict`) for durable task artifacts
